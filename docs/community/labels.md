# Label Taxonomy

Each actionable Issue should normally have one `type:*`, one or more `area:*`,
and one current `status:*` label. Impact labels describe compatibility or user
risk; they do not replace detector severity.

## Type

`type: bug`, `type: feature`, `type: documentation`, `type: research`,
`type: dependencies`, `type: security`

## Area

`area: cli`, `area: installer`, `area: detectors`, `area: crawler`,
`area: evidence`, `area: schemas`, `area: governance`, `area: connectors`,
`area: media`, `area: documentation`, `area: automation`, `area: dependencies`,
`area: release`

## Status

`status: needs-triage`, `status: needs-evidence`, `status: needs-reproduction`,
`status: needs-decision`, `status: ready`, `status: blocked`

## Impact And Contribution

`impact: breaking`, `impact: high`, `impact: moderate`, `impact: low`,
`good first issue`, `help wanted`

Do not create labels for release versions. Use milestones after scope is
accepted. Do not use a label to imply that an idea is approved.
